## 摘要
训练多模态大语言模型（MLLMs）面临着模型和数据异构性的挑战。现有系统通过重新设计训练流程来应对这些挑战，但仍然受限于计算效率和内存效率之间的帕累托前沿，优化一个只能以牺牲另一个为代价。我们提出了BigMac，一种用于多模态大语言模型的新训练流程。BigMac的核心思想是将编码器和生成器的计算优雅地嵌套到原始LLM流程中，形成一个依赖安全的嵌套流水线结构。通过这种设计，BigMac将编码器和生成器的激活内存复杂度降低到O(1)，同时保持LLM的激活内存复杂度不变。同时，它实现了与无限内存的理想化设置相同的计算效率。因此，BigMac打破了计算效率和内存使用之间的帕累托前沿，实现了MLLM训练中计算和内存的同时优化。我们在多个MLLM和训练负载上评估了BigMac。实验结果表明，与基线系统相比，BigMac实现了1.08倍到1.9倍的训练加速，同时随着批次大小增加保持稳定的内存使用。

## Abstract
Training multimodal large language models (MLLMs) is challenged by both model and data heterogeneity. Existing systems redesign the training pipeline to address these challenges, but remain bound by a Pareto frontier between compute and memory efficiency, improving one only at the expense of the other. We present BigMac, a new training pipeline for multimodal LLMs. The core idea of BigMac is to elegantly nest the encoder and generator computation into the original LLM pipeline, forming a dependency-safe nested pipeline structure. With this design, BigMac reduces the activation memory complexity of the encoder and generator to O(1) while keeping the activation memory complexity of the LLM unchanged. At the same time, it achieves the same computational efficiency as the idealized setting with unlimited memory. As a result, BigMac breaks the Pareto frontier between computational efficiency and memory usage, enabling simultaneous optimization of both computation and memory in MLLM training. We evaluate BigMac on multiple MLLMs and training workloads. Experimental results show that BigMac achieves a 1.08$\times$-1.9$\times$ training speedup over baseline systems while maintaining stable memory usage as batch size increases.

- **核心问题**：多模态大语言模型（MLLM）训练面临模型异构性（编码器、LLM、生成器结构差异大）和数据异构性（不同样本的模态token数量差异巨大），导致现有训练系统在**计算效率**与**内存效率**之间存在帕累托前沿——优化计算效率（如Optimus）需要保留大量激活内存，内存使用随批大小线性增长甚至OOM；优化内存效率（如Megatron-DistTrain）则将编码器/生成器嵌入流水线，但引入额外气泡，迭代时间显著增加。
- **整体含义**：现有方法无法同时实现高计算效率与低内存使用，瓶颈在于跨模块的执行干扰和激活生命周期管理。BigMac通过**依赖安全的嵌套流水线**打破这一权衡，实现两者的同时优化，为大规模MLLM训练提供了新路径。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将编码器和生成器的计算**嵌套**到LLM原始流水线中，保持LLM调度不变，仅在依赖满足时插入编码器/生成器操作，从而消除跨模块执行干扰，同时将编码器和生成器的激活内存复杂度降至**O(1)**。
- **关键技术细节**：
  - **流水线调度器**：
    - 将微批次按流水线大小（P）分组为单元（unit），每个单元对应一个LLM流水线列。
    - 预热阶段执行前W=3个编码器前向单元（W=3基于交错1F1B调度的性质：LLM消费单元i的输出后，会在消费单元i+3之前返回单元i的梯度，因此只需3个单元的编码器激活）。
    - 保持LLM原始调度顺序，在每个LLM前向操作后检查依赖：若LLM输出就绪，则执行生成器前向+反向；若编码器梯度就绪（即LLM反向完成对应编码器单元的梯度），则立即执行编码器反向；同时提前准备下一个编码器前向单元。
    - 编码器激活仅需保持3个单元（每个单元P个微批次），生成器激活仅需1个单元（前向+反向连续执行后释放），实现O(1)复杂度。
  - **通信算子**：为每个计算算子派生pre-communication和post-communication，实现跨设备的数据传递；LLM内部使用标准点对点通信；跨模块使用gather/scatter通信。
  - **兼容性扩展**：
    - **上下文并行（CP）**：当LLM和编码器使用不同CP度时，通过异步all-to-all转换布局；调度单元放大到 `P × (LLM_CP_size / Encoder_CP_size)`。
    - **FSDP兼容**：用单向NVSHMEM pull代替all-gather，避免FSDP同步障碍导致的额外气泡，每个rank独立从所有者拉取参数分片。

- BigMac实现了**计算和内存的同时优化**，打破了帕累托前沿。
